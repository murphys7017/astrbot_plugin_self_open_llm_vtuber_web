## 第二部分：核心架构审查

### 1. 插件设计

结论：**入口设计合格，运行时设计不够干净。**

优点：

- [main.py](C:/Users/Administrator/Downloads/AstrBot/data/plugins/astrbot_plugin_self_open_llm_vtuber/main.py#L4) 没有把业务写死在入口

缺陷：

- `plugin_runtime` 使用全局单例状态 [plugin_runtime.py](C:/Users/Administrator/Downloads/AstrBot/data/plugins/astrbot_plugin_self_open_llm_vtuber/adapter/plugin_runtime.py#L12)，这对插件热更新、并行实例、测试隔离都不友好
- 平台适配器本应更像 composition root，但 [platform_adapter.py](C:/Users/Administrator/Downloads/AstrBot/data/plugins/astrbot_plugin_self_open_llm_vtuber/platform_adapter.py#L56) 仍承载过多运行时决策
- 没有真正利用 AstrBot hook 体系做扩展点，更多是“桥接器 + 私有状态机”

### 2. LLM 调用层

结论：**主 LLM 没有被硬编码，这一点好；但表情规划层又额外引入了强串行依赖。**

优点：

- 插件本身不直接写死 OpenAI/Ollama API
- 主对话 LLM 交给 AstrBot 主事件链路
- STT 与表情规划通过 AstrBot Provider 接口获取，[runtime_state.py](C:/Users/Administrator/Downloads/AstrBot/data/plugins/astrbot_plugin_self_open_llm_vtuber/adapter/runtime_state.py#L191) 的抽象方向是正确的

缺陷：

- `expression_provider_id` 本质上引入了**第二个聊天模型调用**，并且在主回复路径上串行执行
- [base_expression_planner.py](C:/Users/Administrator/Downloads/AstrBot/data/plugins/astrbot_plugin_self_open_llm_vtuber/adapter/base_expression_planner.py#L72) 直接依赖 `provider.text_chat()`，没有超时、熔断、预算控制
- 这层没有“慢时降级为规则映射”的明确策略开关，只是在异常时 fallback

### 3. 语音 / VTuber 链路

结论：**链路完整，但不是流式架构，且有多个额外阻塞点。**

输入侧：

- 文本发送前前端会先截图，[use-text-input.tsx](C:/Users/Administrator/Downloads/weather-query/astrbot_plugin_self_open_llm_vtuber_web/src/renderer/src/hooks/footer/use-text-input.tsx#L29)

- 但整体仍然不是 Open-LLM-VTuber 那种“真正流式感知”的链路

## 第六部分：与优秀架构对比

### 1. 对比 AstrBot 官方插件设计

接近的地方：

- `Star` 入口很薄
- 通过 `Context` 和 Provider 接口接入宿主能力
- 配置通过 schema 暴露给宿主

差距：

- 官方风格通常更强调“插件即扩展点”，而当前项目更像“自带 transport + state machine 的边车服务”
- 当前实现使用了 [plugin_runtime.py](C:/Users/Administrator/Downloads/AstrBot/data/plugins/astrbot_plugin_self_open_llm_vtuber/adapter/plugin_runtime.py) 这样的全局运行时容器，不利于插件实例隔离和测试
- 自建 WebSocket/HTTP 服务器让生命周期、权限和部署边界脱离了 AstrBot 统一治理

### 2. 对比 Open-LLM-VTuber 架构

接近的地方：

- 有 VTuber 前端、WebSocket、音频输入、Live2D、VAD 这些典型模块
- 前端和口型链路已经按 OLV 思路改过

## 第八部分：重构路线

### 阶段 1：最小可用优化

1. 统一协议常量，先解决 `interrupt/group-update/heartbeat-ack` 漂移。
2. 去掉默认 `volumes` 计算。
3. 给表情规划加超时与开关，默认规则优先。
4. 把配置读取改成内存缓存。
5. 给 WebSocket/HTTP 加最小 token 鉴权。
6. 为前端输入链路加埋点，量化 `captureAllMedia` 耗时。

### 阶段 2：结构重构

1. 拆分 `TurnCoordinator` 为多个服务类。
2. 引入 `ClientSessionManager`，每连接一个独立 session。
3. 把协议 DTO 抽到共享模块。
4. 去掉 `plugin_runtime` 全局单例，改为实例级依赖注入。
5. 将静态资源服务和 websocket transport 抽象成 `TransportGateway`。

### 阶段 3：高级优化

1. 做真正流式链路：输入流式 ASR、输出流式 TTS、前端分段预播。
2. 将表情规划改为异步 sidecar，不阻塞主回复。
3. 把多模态图片采集改成异步附加上传。
4. 引入 Agent 化扩展点：`EmotionPlanner`、`ActionPlanner`、`MediaPostProcessor`、`FrontendPolicy`。
5. 增加全链路观测：trace id、timing span、回放日志。

## 推荐的核心模块划分

1. `PluginBootstrap`
职责：读取配置、注册 adapter、创建依赖。

2. `TransportGateway`
职责：WebSocket 收发、HTTP 静态资源、鉴权、连接管理。

3. `ClientSessionManager`
职责：按 client/session 管理状态、turn、chat buffer、provider binding。

4. `InboundProtocol`
职责：统一解析 `text-input / mic-audio-data / mic-audio-end / control`。

5. `SpeechIngressService`
职责：音频缓存、VAD、STT、临时文件管理。

6. `AstrBotBridge`
职责：构造 `AstrBotMessage`、提交事件、接收回调消息链。

7. `ReplyPostProcessor`
职责：拆解 Plain/Image/Record、处理多模态输出。

8. `ExpressionPlanner`
职责：规则映射、Provider 规划、超时控制、回退策略。

9. `MediaCacheService`
职责：TTS 文件缓存、图片缓存、资源 URL 生成。

10. `FrontendPayloadEmitter`
职责：把后端域对象转成前端协议 payload。

11. `SharedSchema`
职责：消息类型、DTO、控制码、版本兼容策略。
